📖 Platform Journey & Performance Assessment

Author: Ahmed AlMahdi — Solo Platform Engineer
Period: January 2024 – June 2026
Context: Building and operating a full insurance recovery platform from zero as a solo engineer

Chapter 1: The Foundation (Jan–Jun 2024)

What Was Built

  • Azure infrastructure from scratch using Terraform (modularized)
  • AKS clusters for dev and QA environments
  • 6 .NET 8 microservices scaffolded with clean architecture (API/BLL/DAL/Models)
  • MongoDB on Azure VM with VNet integration
  • Azure B2C tenants configured for authentication
  • Basic CI/CD with GitHub Actions (per-branch deploy)
  • Angular 19 frontends (admin + entity portals)

Challenges Overcome

  • Learning Terraform, K8s, and Azure networking simultaneously
  • No DevOps team — had to be developer + ops + security + DBA
  • Iterative architecture decisions (APIM added later, not initial)

Key Decisions

DecisionRationaleAssessment
MongoDB on VM vs CosmosDBCost savings, full control⚠️ Trade-off (ops burden)
Kustomize over HelmSimpler for 4 envs with small delta✅ Good choice
Microservices from day 1Business domain boundaries clear✅ Appropriate
Branch-per-env deploymentQuick to implement, maps cleanly⚠️ Needs gates for prod

Chapter 2: Scaling Up (Jul–Dec 2024)

What Was Added

  • Staging and Production environments
  • Azure API Management (APIM) for all environments
  • Azure Application Gateway with WAF for production
  • Key Vault integration via CSI driver (QA/Staging/Prod)
  • Redis Cache for session and data caching
  • TriggerHub (Azure Functions) for background processing
  • Grafana + Prometheus monitoring stack (prod)

Growing Pains

  • Config drift between environments (fixed June 2026)
  • APIM rate limiting tuning took iterations
  • B2C token refresh implementation complexity
  • Managing 4 parallel environments solo

Chapter 3: Production Operations (Jan–Jun 2025)

Key Milestones

  • Platform went live for initial pilot users
  • Implemented availability tests and alerting rules
  • Added Azure Firewall with threat intelligence
  • Deployed Bastion for secure administrative access
  • CosmosDB for payment processing
  • SignalR (WebSocket) for real-time notifications

Chapter 4: Maturity & Hardening (Jul 2025 – Jun 2026)

Current Focus

  • Documentation (this suite of docs)
  • Environment consistency (config standardization)
  • CI/CD improvements (DRY workflows)
  • Planning for autoscaling and GitOps
  • Security hardening gaps identified and prioritized
